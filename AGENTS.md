# AGENTS.md

本文件是给 AI 编程代理看的项目指引，作用范围为当前目录及其所有子目录。

## 项目定位

这是一个本地优先的保险客户工作日历，不是后台管理系统。目标用户是单个保险销售顾问。用户每天最关心的是：今天该联系谁、为什么联系、保费是多少、点完成后有没有真的记录完成。

界面要尽量少解释、少配置、少暴露底层概念。不要把数据库字段、同步 token、表结构、统计底册放到日常主界面里占空间。

## 当前产品决策

- 工作日历是默认主入口。
- 月历中已经有年月、上月、下月、今天，因此顶部工具条不再重复这些控件。
- 顶部生日、续期、待办数量按当前月份计算。
- 生日、续期、待办必须有稳定颜色区分：生日粉色，续期蓝色，待办绿色。
- 右侧当天详情卡片不重复写“生日/续期/待办”标签，颜色已经承担分类。
- 续期卡片预览只显示客户姓名和保费，点开后展示具体保单详情。
- 点击完成必须更新本地数据库状态，不能只是视觉跳一下。
- 当天详情区需要支持“完成当天全部”。
- 导入入口保持集中，不要在多个区域重复放“导入”按钮。
- 新增待办入口保持集中，工作台已有入口时不要再在侧边栏重复加。
- 大块统计、数据健康、底册数量属于低频信息，应折叠、变小或移到低频页面，不要占据工作日历主视觉。

## 开发原则

- 如无必要，不新增实体按钮、导航入口或常驻控件。
- 如果确实需要新增实体按钮，必须在说明中写清楚理由，并从两个角度解释：它如何改善客户体验；它如何帮助保险从业者开展业务。
- 优先删除、合并或复用已有入口。不要为同一件事提供多个相似入口。
- 每个功能尽量保持独立，避免为了修 A 改坏 B。领域规则、数据库、API、UI、同步逻辑要按边界分层。
- 不要把低频配置暴露到高频工作界面；高频界面只放业务人员每天真的会用的动作。

## 飞书同步 UX

飞书同步页要面向非技术用户：

- 第一输入框表达为“飞书链接粘这里”。
- 用户可以粘贴 `https://xxx.feishu.cn/base/...` 这样的多维表格链接，也可以粘贴直接 Base 标识。
- 这个链接必须是飞书多维表格，且当前账号要有编辑权限。
- 链接本身不等于授权；本机还需要完成 `lark-cli` 飞书授权。
- 不要要求普通用户理解 Base token、table id、字段 schema、batch create 等术语，除非放在高级/调试区域。
- 同步执行前优先提供计划和可读的错误信息。

## 架构边界

- `src/domain/`：纯业务规则，不读文件、不访问 SQLite、不调用飞书、不依赖 React。
- `src/importers/`：读取和归一化 Excel，可理解源文件列名，不直接写数据库或 UI。
- `src/db/`：数据库连接、迁移和仓储。
- `src/api/`：本地 HTTP API，负责把 UI 操作连接到仓储、导入和同步服务。
- `src/sync/`：飞书多维表格和飞书日历同步，只使用本地快照，不重新计算提醒。
- `src/web/`：React UI，只通过 `apiClient.ts` 调 API，不直接导入数据库、导入器或领域服务。
- `tests/contracts/`：跨模块行为合同测试。

## 业务规则

- 客户识别优先使用姓名 + 证件号严格匹配。
- 完整证件号和脱敏证件号匹配时，必须姓名相同且前后位同时匹配。
- 不确定的身份匹配进入待确认，不要自动合并。
- 续期只相信明确缴费期间字段，不从产品名或经验猜缴费年限。
- 支持固定年期和年龄型缴费期间；缺生日、生效日或无法解析时进入待确认。
- 已完成的提醒不应继续作为未处理工作项展示。
- 手动待办可以编辑和删除；导入生成的生日/续期提醒不能被当作普通手动待办删除。

## 数据安全

- `data/customer-reminders.sqlite` 和 `data/backups/` 是本地敏感数据，不要提交。
- 不要在日志、文档、测试快照或飞书同步预览里输出完整手机号、完整证件号、Base token 或 session cookie。
- 客户外部 ID 必须保持 hash 形态，不要恢复为 `customer:姓名:证件号`。
- 处理隐私相关改动时，检查数据库、API、UI、飞书同步、多维表格字段、日历描述和测试。
- 任何新增外部读取、写入或同步能力，都必须先定义数据权限、脱敏输出和用户可理解的授权边界。

## GitHub 开源发布硬规则

- 凡是准备进入公开 GitHub 的开源发布、公开 PR 或公开分支，不得直接从私有开发区 `/Users/mac/Documents/小程序/客户提醒` 提交和推送。
- 必须先运行白名单导出：`npm run export:open-source -- --clean`，把可公开内容导出到公开发布区 `/Users/mac/Documents/开源/AI保顾问`。
- 必须在公开发布区重新执行验证：`npm install`、`npm run typecheck`、`npm test`、`npm run build`、`npm audit --omit=dev`，并做敏感内容扫描。
- 敏感扫描至少确认没有 `data/`、`data/backups/`、`.omx/`、`示范文件/`、SQLite、真实 Excel/PDF、真实飞书链接、Base token、坚果云账号或应用密码、session cookie、客户截图、私人二维码或内部品牌素材。
- 验证后要清理公开发布区的本地产物：`node_modules`、`dist`、`.omx`，再检查 `git status -sb`。
- 只有公开发布区验证和扫描通过后，才可以从公开发布区提交、推送 GitHub 或创建 PR。
- 如果只是临时私有备份分支，必须明确说明“未走开源发布流程”，不得称为“已开源发布”。

## 前端修改要求

- 先看当前页面结构和现有 CSS，不要直接堆新卡片。
- 不要新增重复入口；已有入口能完成任务时，优先删除或合并。
- 文字要短，面向业务用户，不写“系统将会”“字段 schema”等技术说明。
- 卡片、按钮、筛选器和弹窗要在桌面和窄屏下都不重叠。
- 日历相关改动要验证：月份切换、今天按钮、日期选择、事件点击、右侧详情、完成状态。
- 完成/恢复/批量完成这类操作必须真实调用 API 并刷新状态。

## 后端和同步修改要求

- Fastify POST 请求如果声明 JSON，前端不能发送空 body 加 JSON content-type；空操作也要明确发送 `{}` 或不设置 JSON 头。
- 飞书 Base 同步要支持从完整链接解析 Base 标识。
- 执行飞书写入前保留 plan 模式，便于用户确认。
- 批量创建要记录 `sync_state`，重复执行不能重复创建同一外部 ID。
- 增量更新优先使用已保存的飞书 record id。

## 文档维护

- `README.md` 是给人看的全局入口，保持当前能力、启动方式、产品原则和安全说明同步。
- `SPEC.md` 记录业务规则和成功标准，修改核心规则时同步更新。
- 新增重要迁移或大规模验收流程时，放到 `docs/` 并从 README 链接。
- 不要把私有商业规划、历史迭代记录、真实客户资料或未公开路线写入公开仓库。
